INTERVIEWER: Let's start at the ground floor. Walk me through the difference between primitive and reference types, and while you're in there, settle the old debate for me, is Java pass-by-value or pass-by-reference?

CANDIDATE: Java is always pass-by-value, and the primitive versus reference split is exactly why people get that wrong. A primitive like int or boolean holds the value directly, so assignment copies the value, if I copy an int into a second variable and change the copy, the original stays untouched. A reference type, meaning objects, arrays, and String, holds a reference to data on the heap, so assignment copies the reference and both variables end up pointing at the same object. That carries into method calls, when I pass an object in, the reference itself is copied, so the method can change the object's fields but it can never reseat my variable to point somewhere else, which is why it is still pass-by-value. The classic trap is copying an array, changing element zero through the copy, and being surprised the original changed too, both variables share the same array on the heap.

INTERVIEWER: Nicely grounded, that reseating point is the one most people miss. Now, Strings trip candidates up constantly, so tell me the difference between double equals and the equals method, and why String is immutable in the first place.

CANDIDATE: Double equals compares references, asking whether two variables point at the same object, while equals compares content, so for Strings and objects generally I always compare with equals. The reason double equals sometimes appears to work is the string pool, identical literals are interned and shared, so two variables assigned the same literal actually point at one pooled object and double equals happens to return true, but build the same text with new String and it returns false even though the content matches, which is a nasty trap in test assertions. On immutability, a String never changes once created, every so-called modification hands back a new object, and that buys thread-safety, a stable hashCode so Strings are safe as HashMap keys, and the pool sharing itself. The practical consequence is that concatenating in a loop allocates a fresh object every pass, so for building output row by row I reach for StringBuilder and do one final allocation at the end.

INTERVIEWER: Good, and you walked straight into my next one. You mentioned hashCode, so explain the equals and hashCode contract and what actually breaks when you violate it.

CANDIDATE: The contract says that if two objects are equal they must return the same hashCode, and the reason is how hash collections work, a HashMap or HashSet locates an object by its hash first to pick the bucket, and only then uses equals to confirm the match. So if I override equals without hashCode, two logically equal objects can land in different buckets, lookups miss, entries effectively go missing, and duplicates slip into sets. I hit this on a real project, we placed expected user response objects into a Set and checked whether each actual API response was contained in it, and every contains call came back false even with perfectly matching data, because the class overrode neither method and the default reference equality kicked in. Generating equals and hashCode together from the same fields fixed the test immediately, and in modern Java a record generates both for you, which makes records a great fit for test data objects.

INTERVIEWER: That is exactly the failure mode, silent and confusing. Let's move to design. Abstract class versus interface, what's the difference, and where does each one live in a test framework you'd build?

CANDIDATE: An abstract class is a partial base with shared code and state, a class extends exactly one of them, and it models an is-a relationship, while an interface is a contract, a capability, a class can implement many, and since Java 8 an interface can even carry default methods. The deciding question is what I'm actually sharing, if subclasses genuinely share fields and setup logic, an abstract class holds that in one place, but if I only need to promise a capability across otherwise unrelated classes, an interface does it without forcing everyone into a hierarchy. In a Selenium framework that maps very cleanly, a BasePage abstract class holds the WebDriver reference and the common waiting logic, and every page object extends it, while something like a Navigable interface marks just the pages that support direct URL navigation so tests can drive those uniformly. The trap is using deep inheritance to share code, composition is usually the better tool there.

INTERVIEWER: Solid, I like that you tied it to page objects. Staying practical, when you're writing a test helper, how do you choose between an ArrayList, a HashSet, and a HashMap?

CANDIDATE: I ask three questions, does order matter, are duplicates meaningful, and which operation dominates. ArrayList keeps insertion order, allows duplicates, and gives fast index access, so it fits an ordered sequence like test steps or a request log where position carries meaning. HashSet enforces uniqueness with roughly constant-time contains, so it fits membership checks and duplicate detection. HashMap is my default for key to value lookup, counting word frequency with getOrDefault is the classic one-pass example. The performance trap is calling contains on an ArrayList inside a loop, every check scans the whole list, and I lived that one, an API response validator kept expected field names in a list and checked each against the actual fields with contains, and for large schemas it crawled, switching the actual fields to a HashSet cut validation time roughly twenty times. And when I need uniqueness but the order still matters, LinkedHashSet gives me both, it drops duplicates while preserving insertion order.

INTERVIEWER: Good instincts, that contains-in-a-loop mistake is everywhere. Two more. Checked versus unchecked exceptions, and how do you keep exception handling clean in test code?

CANDIDATE: A checked exception must be handled or declared with throws, the compiler enforces it, IOException is the classic case, while unchecked means the RuntimeException family, things like NullPointerException or IllegalArgumentException, where nothing is enforced. The idea is that recoverable external failures, files, streams, that sort of thing, get acknowledged at compile time. My hygiene rules are to catch the narrowest exception, never leave an empty catch block, and keep the original cause whenever I wrap. So reading a data file I'd use try-with-resources, which automatically closes anything AutoCloseable even when an exception is thrown mid-read, and if the read fails I wrap the IOException in a RuntimeException with the cause attached, so the stack trace still tells the whole story. In tests specifically I let unexpected errors fail loudly, because a swallowed exception is how a broken test quietly passes, and a test that lies is worse than no test at all.

INTERVIEWER: Couldn't agree more on failing loudly. Last one then. What is a stream, and just as important, when would you deliberately not use one?

CANDIDATE: A stream, from Java 8, processes a collection as a pipeline, a source, then lazy intermediate operations like filter and map, then exactly one terminal operation like collect or count, and nothing executes until that terminal operation runs, the intermediate steps just describe the work. The win is readability for filter, transform, and collect jobs, a concrete case from API testing is filtering a list of response rows to keep only the ones where the status field equals active, streaming the rows, filtering with Objects dot equals so an absent key returning null doesn't blow up with a NullPointerException, then collecting to a list, and Collectors groupingBy does the same kind of heavy lifting when I need rows bucketed by a field. But I don't force everything into a stream, a plain for loop is fine and often faster for simple iteration or when I need to break early, and the beginner trap is returning the stream itself without a terminal operation, so the pipeline never actually runs.
