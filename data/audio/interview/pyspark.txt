INTERVIEWER: Let's start at the foundation. Walk me through how Spark actually runs a job, and why people keep saying it's lazy.

CANDIDATE: Spark is a distributed engine, so there's one driver that plans the work and many executors that run it in parallel, with the data split into partitions and one task processing one partition. The lazy part means transformations like select, filter, or join don't run anything, they just build up a plan, and nothing executes until an action like count, show, or write asks for a result. The why is optimization, because deferring execution lets the Catalyst optimizer see the whole plan and rewrite it, pushing filters down, pruning columns, reordering joins, before a single row moves. A concrete trap I've hit in testing is calling collect on a big DataFrame, which pulls every row back to the driver and can crash it with an out of memory error, so in tests I assert on aggregates or a bounded limit instead of collecting everything.

INTERVIEWER: Good, and you already flagged the collect trap, which most people miss. Now, PySpark specifically, how do you think about RDD versus DataFrame versus Dataset?

CANDIDATE: The direct answer is that in PySpark you use DataFrames, almost always. A DataFrame is a distributed table with a schema, named and typed columns, and because Spark knows the structure, Catalyst can optimize your queries and Tungsten can generate efficient code. An RDD is the low level distributed collection underneath, full control but no schema and no optimizer, so it's reserved for rare cases like custom partitioning or truly unstructured data. And the Dataset, which adds compile time type safety, only exists on the JVM in Scala and Java, so in Python the whole DataFrame versus Dataset debate collapses to just use DataFrames. One habit worth mentioning is that when I read data I always pass an explicit schema rather than using inferSchema, because inference reads the file twice and can silently change types between runs, which is a flaky test factory.

INTERVIEWER: That schema point is one I wish more candidates made. Let's talk performance. What is a shuffle, and how do repartition and coalesce fit into that picture?

CANDIDATE: A shuffle is Spark redistributing data across executors for a wide transformation like groupBy, join, or distinct, and it's the single most expensive thing Spark does, because it writes intermediate data to disk and reads it back over the network. That's why the narrow versus wide distinction matters, a narrow transformation like filter or select stays within a partition and is cheap, while wide ones regroup rows across the cluster. Repartition does a full shuffle to exactly the number of partitions you ask for, up or down, and you use it to fix skew or raise parallelism. Coalesce only merges partitions down and avoids the full shuffle, so it's the cheap option right before a write, which prevents the small files problem where you spray thousands of tiny files into storage. And post shuffle partition count defaults to two hundred, which adaptive query execution now tunes at runtime.

INTERVIEWER: Nice. You mentioned skew, so let's go there. Your join has a hundred and ninety nine tasks finishing in seconds and one running for an hour. What's happening and what do you do?

CANDIDATE: That's the classic signature of data skew, a few hot keys, maybe one heavy user id or a null join key, holding most of the rows, so one task gets a giant partition and the whole stage waits on it. My fixes go in order of preference. First, enable adaptive query execution's skew join handling, because Spark will split the skewed partitions automatically and it's the cheapest thing to try. Second, if the small side of the join fits under the broadcast threshold, around ten megabytes by default, broadcast it to every executor so the join happens locally with no shuffle at all, and no shuffle means no skew. Third, salting, where I append a random suffix to the hot key on both sides to spread it across partitions and strip it after the join. And for null or sentinel keys, I'd handle those rows separately and union the results back in.

INTERVIEWER: Solid ladder of options. Here's a practical one you'd hit in any pipeline. You have duplicate records per key and you need to keep only the most recent one. How do you do that?

CANDIDATE: That's a window function job. I'd define a window partitioned by the key and ordered by the timestamp descending, apply row number over it, and keep only the rows where the row number equals one, then drop the helper column. The why is that window functions compute across related rows without collapsing them the way groupBy does, so I get to rank every record within its key and choose the winner, and the descending order guarantees the latest one ranks first. It's the same machinery I'd use for running totals with sum over a window, or period comparisons with lag and lead. A trap worth calling out is reaching for dropDuplicates and assuming it keeps the most recent row, it doesn't make that guarantee, so the row number pattern is the reliable answer. It's also a lovely unit test, two rows for one id with different timestamps in, one row out, the newer one.

INTERVIEWER: And you just wrote your own test case for it, which is a good segue. As an SDET, the pipeline ran green last night. How do you know the data is actually right?

CANDIDATE: Green just means the code didn't crash, so I test the data itself against a checklist of quality dimensions. Completeness, no unexpected nulls in required columns. Uniqueness, primary keys aren't duplicated. Row count reconciliation, the output count squares with the source, because an inner join can silently drop rows and nobody notices. Range and domain checks, amounts are non negative and statuses come from an allowed set. Referential integrity, every foreign key resolves to a dimension row, and freshness, the latest partition is recent enough. The mature pattern is to quarantine rather than crash, routing bad rows to a dead letter table so good data keeps flowing while the bad rows are captured for triage. And when a run looks wrong on a Delta table, time travel is my favorite debugging tool, I diff the current version against the previous good version to isolate exactly which rows changed.

INTERVIEWER: That quarantine versus crash distinction is exactly the maturity signal I'm listening for. Last one. How do you unit test PySpark transformations so this all runs in CI?

CANDIDATE: The trick is structure. I keep the logic in pure transform functions that take a DataFrame and return a DataFrame with no input or output inside them, which makes the pipeline testable like any other code. In pytest I spin up a session scoped local SparkSession using local star, build tiny inputs with createDataFrame, and run the function. For assertions I use a proper DataFrame equality helper, chispa's assert df equality or the built in assertDataFrameEqual from Spark three five onward, rather than comparing collected lists, because those helpers handle column order and nullability and give readable diffs when something breaks. On top of the unit layer, the data quality checks I mentioned become contract tests, running through something like Great Expectations or dbt tests as a gate in CI. And the earlier trap still applies, never assert on a full collect of production scale data, keep test inputs small and assertions on aggregates.
