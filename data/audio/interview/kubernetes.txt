INTERVIEWER: Let's start at the top. You already know Docker, so tell me, what does Kubernetes actually add on top of Docker, and why would a team need it?

CANDIDATE: The direct answer is that Docker runs a container on a host, while Kubernetes orchestrates many containers across a whole fleet of machines, handling scheduling, self-healing, scaling, and rollouts. The reason that matters is the model it uses, you declare desired state, say three replicas of this image, and Kubernetes continuously reconciles reality to match that declaration. So if a Pod crashes it gets recreated, if a node dies its Pods get rescheduled somewhere else, and if load spikes you can add replicas by hand or automatically. My mental model is that Docker is the package and Kubernetes is the operations team that runs those packages and keeps them alive. A concrete example, when we ship a new version, Kubernetes rolls it out gradually with zero downtime, and if it turns out bad we revert instantly, which is something plain Docker on a host simply doesn't give you.

INTERVIEWER: Good, and I like that you led with the desired state idea. So on workloads then, what's the difference between a Pod and a Deployment, and which one do you actually write?

CANDIDATE: A Pod is the smallest deployable unit, one container or a few tightly coupled ones sharing a network, an IP, and storage, and the key property is that Pods are ephemeral, they come and go with new IPs. A Deployment sits above that, it manages a ReplicaSet that keeps N identical Pods running, and it gives you rolling updates and rollbacks. The why is exactly that ephemerality, if I created bare Pods and one died, nothing would bring it back, whereas a Deployment's controller notices the gap between desired and actual state and recreates it. So in practice you almost never create Pods directly, for a stateless app you write a Deployment with, say, replicas set to three and a label like app web, and it manages the Pods and their replacement for you. The trap is treating a Pod as something permanent, it isn't, and designing around that assumption breaks fast.

INTERVIEWER: Right, and that ephemerality leads nicely into networking. If Pods keep changing IPs, how does anything ever talk to them reliably?

CANDIDATE: That's what a Service is for. Since Pods are ephemeral with changing IPs, you never talk to a Pod directly, instead a Service gives you a stable virtual IP and DNS name and load-balances across the matching Pods. The clever part, the why, is that the Service finds its Pods by label selector, not by IP, so as Pods get replaced during a deploy or a crash, the Service keeps working without anyone updating addresses. For exposure levels, ClusterIP is the default and is reachable only inside the cluster, NodePort opens a fixed port on every node, and LoadBalancer provisions an external cloud load balancer. And in front of Services you often put an Ingress, which is an HTTP router doing host and path rules plus TLS termination, so many apps share one external entry point instead of paying for a load balancer each. A common trap is exposing everything as LoadBalancer when Ingress is the cleaner answer.

INTERVIEWER: Solid. Now here's the classic one, and people mix this up all the time, liveness versus readiness probes, what's the difference?

CANDIDATE: The direct answer, a liveness probe asks whether the container is alive, and if it fails Kubernetes restarts the container, while a readiness probe asks whether it can serve traffic, and if it fails the Pod is only pulled out of the Service's load balancer, it is not killed. The reason both exist is that they catch different failure modes, liveness catches things like deadlocks where the process is stuck and only a restart helps, readiness catches warm-up time or waiting on a dependency, situations where the app will recover on its own and just shouldn't get traffic yet. A concrete setup would be a readiness probe doing an HTTP get on a healthz endpoint with an initial delay of a few seconds. The trap you called out is real, if you mix them up, say you put a dependency check in liveness, a slow database sends your app into a restart loop instead of just sitting out of rotation.

INTERVIEWER: Exactly, that restart loop is a war story for a lot of teams. Let's talk configuration, how do you get config and credentials into your app, and what's the catch with Secrets?

CANDIDATE: You use a ConfigMap for non-sensitive configuration, things like URLs and feature flags, and a Secret for sensitive data like passwords, tokens, and TLS keys, and both get injected as environment variables or mounted as files. The why is separation of config from the image, the same image runs in dev, staging, and test namespaces with different settings, so you're testing the exact artifact you'll ship. In a manifest that looks like an env var whose value comes from a configMapKeyRef or a secretKeyRef. The catch, and this is the interview trap worth calling out yourself, is that a Secret is only base64-encoded by default, not encrypted, so anyone who can read the object can decode it. In production you'd add RBAC to control access, enable encryption at rest, or use an external secret store like Sealed Secrets or Vault rather than trusting base64 to protect anything.

INTERVIEWER: Good, you beat me to the base64 point. Now walk me through shipping a new version, how do you get zero downtime, and what keeps one Pod from eating the node?

CANDIDATE: The default strategy is a rolling update, Kubernetes replaces Pods gradually, so with kubectl set image on the deployment it brings up new Pods, waits for them to pass readiness so they only receive traffic when they're ready, and retires the old ones, meaning there's never a moment with nothing serving. You can watch it with kubectl rollout status, and if the new version is bad, kubectl rollout undo reverts to the previous version instantly, which is the safety net that makes frequent deploys sane. On the resource side, requests and limits are the guardrails, requests tell the scheduler how much CPU and memory to reserve when placing the Pod, and limits cap actual usage. The trap to know is that a container going over its memory limit gets OOM-killed, so if you see containers dying with no application error in the logs, a too-tight memory limit is one of the first things I'd check.

INTERVIEWER: Nice, and that OOM point sets up my last question. As an SDET, where do you actually touch Kubernetes day to day, and what's your first move when a test deployment misbehaves?

CANDIDATE: The biggest win for me is ephemeral test environments, you spin up a namespace or even a throwaway cluster per branch or pull request, deploy the app, run the end to end suite, and tear it all down, and locally or in CI you can run a real cluster with kind or minikube. Helm helps here too, templated versioned charts mean the same deploy is parameterized across dev, test, and prod instead of drifting apart. When something misbehaves, my triage toolkit is kubectl, get pods to see what's running, describe pod to read the events, logs to see app output, and exec or port-forward to poke at the service directly. The first move specifically, if a Pod is stuck in Pending or CrashLoopBackOff, is kubectl describe pod, because the events section almost always names the cause, no node fits, the image pull failed, or a probe is failing. Starting there saves you from guessing.
