INTERVIEWER: Welcome in, good to have you here today. Let's run a quick Playwright round, the kind of rapid technical screen you'd get for an SDET role. I'll ask a question, you give me your best answer, and I may push a little deeper here and there. Nothing exotic, just the fundamentals done well. Ready when you are.

CANDIDATE: Sounds good, let's get into it.

INTERVIEWER: Let's start at the top. You've used Selenium before, so tell me, what actually makes Playwright different, beyond it just being newer?

CANDIDATE: The core difference is the protocol and what's built in. Playwright talks to the browser over a persistent DevTools style connection instead of the WebDriver wire protocol, so it's one process holding an open channel to Chromium, Firefox, and WebKit through a single API. Why that matters is speed and reliability, there's no request and response handshake per command, and Playwright layers auto-waiting into every single action, so the flaky wait logic you'd assemble yourself in Selenium comes for free. On top of that you get network interception, tracing, video, and screenshots without bolting on third party tools. A concrete example, in a Selenium suite I'd maintain a custom explicit wait helper for every dynamic element, and in Playwright that whole helper class disappears because a click already waits for the element to be visible, stable, and enabled. The trap is assuming they're interchangeable, the architecture is genuinely different.

INTERVIEWER: Good framing. You mentioned the browser once, so let's go there, walk me through the browser, context, and page model.

CANDIDATE: Three layers. The browser is the actual instance, it's expensive to launch, so you reuse one across the whole run. A browser context is an isolated session inside that browser, its own cookies, storage, and cache, essentially a fresh incognito profile, and it's cheap to create. A page is a single tab inside a context, and that's where most interactions happen. The why is the trade-off between speed and isolation, relaunching a browser per test would be clean but painfully slow, while sharing one session would leak state between tests. Contexts give you both, one browser, but every test gets its own clean context, so no test sees another test's cookies or login. In practice that means a test that logs in as an admin and a test that checks the logged out view can run back to back, or even in parallel, without ever polluting each other, and that context isolation is exactly what makes safe parallel execution possible.

INTERVIEWER: Nice, and you flagged the interview point yourself, that isolation without relaunching is the key. Next one, how do you choose locators, and what is auto-waiting actually doing for you?

CANDIDATE: My first choice is getByRole, locating by the ARIA role and accessible name, because it matches how a user and assistive technology perceive the page, so it survives markup refactors that would kill a CSS class or an id selector. After that I reach for getByLabel, getByText, or a stable data-testid hook, and CSS or XPath only as a fallback. The deeper point is that a locator is lazy, it re-queries the DOM every time you act on it, which is why stale element problems basically vanish in Playwright. Auto-waiting means that before any action fires, Playwright waits for the element to be attached, visible, stable, enabled, and able to receive events, and web-first assertions like expecting a locator to be visible keep retrying until the condition holds or times out. The trap I call out in reviews is someone grabbing the text content once and asserting on that snapshot, that races on slow machines, whereas the retrying assertion just waits it out.

INTERVIEWER: That distinction between a one-shot snapshot and a retrying assertion trips up a lot of people, well spotted. Now, logging in through the UI before every test gets expensive fast. How do you handle authentication across a big suite?

CANDIDATE: The pattern is authenticate once and reuse the state. In a global setup step I log in through the app one time, then save the storage state, which is the cookies plus local storage, out to a JSON file. The config then points new contexts at that file, so every test starts with the session already active and never touches the login screen. The why is both speed and stability, login is usually the slowest and one of the flakiest flows in any app, so removing it from hundreds of tests is a massive win, it's the equivalent of seeding a logged in session instead of clicking through the form each time. On one suite that alone shaved minutes off every CI run. It also extends naturally to multiple roles, you save one state file per role, admin, editor, viewer, and a test that needs two roles just opens two contexts, each loading its own file. The trap is doing both, loading saved state and also running a login fixture, which double authenticates and can create session conflicts.

INTERVIEWER: Solid, that's the pattern I wanted to hear. Let's talk about the network. Your UI test depends on a backend that's slow or flaky, what do you do?

CANDIDATE: I isolate the UI from the backend with network interception. Playwright lets me register a route against a URL pattern, and inside that handler I can fulfill the request with fixed JSON of my own, modify it, or abort it entirely to simulate a network failure. So for a deterministic UI test I stub the API, the users endpoint returns exactly the payload I define, every run, and now the test is verifying my rendering logic rather than the backend's mood that day. The why is determinism, a UI test should fail because the UI broke, not because staging was slow. A concrete case, testing an empty state screen used to mean somehow arranging an account with no data, instead I intercept the list endpoint and fulfill it with an empty array, and the empty state renders instantly. When I do need the real backend, I can wait for a specific response before asserting. The trap is over-mocking, if everything is stubbed you're no longer testing integration, so I keep a thin layer of end to end tests against the real API alongside the mocked ones.

INTERVIEWER: Good balance at the end there, mocking is a tool, not a religion. Last one, and it's the practical one. A test passes locally every time but fails intermittently in CI. Walk me through how you attack that.

CANDIDATE: My first stop is the trace, not re-running the test. I configure tracing to record on the first retry, so when CI retries a failed test, Playwright captures a full trace, DOM snapshots before and after every step, the network waterfall, and console output. I open that in the Trace Viewer and step through the failure frame by frame, which usually shows me a timing issue, an element momentarily detached during a re-render, or an API call that takes ten times longer on the CI box than on my machine. The why is that CI flakiness is almost always a race the faster local environment hides, so I need evidence from the failing environment, not a local reproduction that will never fail. A real example, a continue button that was briefly disabled while a validation call finished, locally the call was instant, in CI it wasn't, and the fix was asserting the button was enabled before clicking, letting auto-waiting absorb the delay. The trap is just bumping the timeout, that hides the race instead of fixing it, and retries should be a safety net, not the cure.
