INTERVIEWER: Let's start with the classic. Node is famously single-threaded, so how does it handle thousands of concurrent I/O operations without blocking?

CANDIDATE: The direct answer is that only your JavaScript runs on a single thread, while the actual I/O is offloaded to libuv, the C library that gives Node its event loop and a background thread pool. The reason that works is the model itself, single-threaded, non-blocking, and event-driven, so when you kick off a file read or a network call, Node hands that work to libuv or the operating system, keeps running your code, and when the operation finishes, the callback gets queued for the event loop to pick up. Node itself runs on the V8 engine for the JavaScript, and libuv handles files, network, and timers in the background. A concrete example, an HTTP server can juggle thousands of open connections on one thread because none of them ever block it. The trap is assuming single-threaded means one thing at a time, that's only true for your own JavaScript, not for the I/O underneath.

INTERVIEWER: Good, and you mentioned the event loop, so let's go there. Walk me through how the loop is structured, and where microtasks fit in.

CANDIDATE: The event loop runs in phases, in a fixed order, and each phase drains its own queue. First timers, where setTimeout and setInterval callbacks fire, then pending callbacks for deferred I/O, then poll, which retrieves new I/O events and runs their callbacks, then check, which is where setImmediate lives, and finally close callbacks, like a socket closing. The part interviewers really care about is what happens between phases, because that's where microtasks run. Process next tick fires first, then resolved Promise callbacks, and both queues drain completely before the loop moves on to the next phase. That ordering matters in practice. The trap to call out is a runaway next tick, if you keep scheduling next tick callbacks recursively, the loop never gets back to the poll phase, and you literally starve your I/O. So next tick is powerful, but it jumps the queue ahead of everything, and you have to respect that.

INTERVIEWER: Nicely laid out. Now something you hit the moment you set up a test project, CommonJS versus ES modules. What's the actual difference, and how does Node decide which one a file is?

CANDIDATE: CommonJS is the original system, require and module dot exports, and it loads synchronously. ES modules use import and export, they're statically analyzable, they load asynchronously, and they support top-level await, which CommonJS doesn't. The deciding factor for a bare dot j s file is the type field in package json, if it says module, plain files are treated as ES modules, otherwise they default to CommonJS, and you can always force it per file with the dot m j s or dot c j s extensions. Why it matters for an SDET is very practical, half the debugging pain in a new test framework is a module mismatch, some tool published as ESM being pulled into a CommonJS config. The interop rule to remember is that ESM can import CommonJS fine, but CommonJS historically could only load an ES module through dynamic import, though recent Node versions added a synchronous require for ES modules. Knowing that saves you from a very confusing error message.

INTERVIEWER: That's exactly the practical angle I wanted. Next up, async patterns. Give me the evolution, callbacks to Promises to async await, and how you'd actually use them well.

CANDIDATE: There are three generations, each built on the last. The original convention is the error-first callback, where the function takes a callback whose first argument is always the error, and nesting a lot of those gives you callback hell. Promises fixed that, an object representing a future value that you chain with then and catch, and they compose, so Promise all, all settled, and race let you coordinate concurrent work. Then async await is syntactic sugar over Promises, you write asynchronous code that reads sequentially and use a normal try catch for errors, and that's the modern default. The why is readability and error handling, flat sequential-looking code is far easier to reason about in a test suite. Two practical tips I'd add, wrap old callback APIs with util promisify or just use the promise variants of core modules like f s promises, and here's the trap, if you have independent async work, run it concurrently with Promise all instead of awaiting each call in sequence, because sequential awaits silently serialize things that could run in parallel.

INTERVIEWER: Good catch on the sequential await trap, that one shows up in slow test suites all the time. Let's talk streams. What are they, and why would you reach for stream pipeline instead of chaining pipe calls?

CANDIDATE: A stream processes data in chunks instead of loading everything into memory, which is essential for large files and network data. There are four types, readable, which is a source like a file read stream or an incoming HTTP request, writable, a destination like a file or an HTTP response, duplex, which is both, like a TCP socket, and transform, a duplex stream that modifies data passing through, gzip or encryption being the classic examples. Underneath, the chunks being moved around are Buffers, raw binary data. Now, to the pipeline question, the direct answer is error handling and cleanup. Chained pipe calls don't propagate errors across stages, so a failure mid-chain can leak resources, while stream pipeline forwards errors from any stage and destroys all the streams properly on failure. Both preserve backpressure, meaning when the writable can't keep up, the readable gets paused automatically so memory stays bounded. A real scenario, gzipping a huge log file, read stream into a gzip transform into a write stream, pipeline handles the whole chain in one await.

INTERVIEWER: Solid. Now error handling more broadly, because Node has several async styles, each with its own failure mode. How do you handle errors across them, and what's your take on uncaught exception handlers?

CANDIDATE: The rule is that the handling strategy follows the async style. Synchronous code and async await use try catch, callbacks mean checking that error-first argument every single time, Promises use catch, and if you miss one, the process emits an unhandled rejection event you can hook. Then there's the EventEmitter family, streams, HTTP servers, sockets, they're all emitters, and the trap there is famous, an error event with no listener attached doesn't get ignored, it's thrown and can crash the whole process, so any emitter that can fail needs an error listener. On uncaught exception, my position is that it's a last resort, never a recovery tool. After one fires, the process is in an undefined state, so the right move is to log, run only the critical cleanup, and exit, then let a process manager like p m two, systemd, or Kubernetes restart the process cleanly. Catching it and carrying on as if nothing happened is how you end up serving requests from a corrupted state.

INTERVIEWER: That's the mature answer, exit and let the supervisor restart. Last one. Your Node process is pegging one core, or the event loop is getting blocked by heavy computation. What are your options for scaling, and how do you choose?

CANDIDATE: The direct answer is that the choice follows the kind of work. If the service is I/O-bound and you just need more throughput, use the cluster module, it forks multiple Node processes that share one listening port, the primary accepts connections and hands them to workers, round-robin by default on Linux and macOS, and that spreads an I/O-heavy server across all your cores. If the problem is CPU-bound work blocking the event loop, parsing huge files, image processing, that kind of thing, cluster doesn't help a single heavy computation, you want worker threads, real threads inside one process that can even share memory through a SharedArrayBuffer, so the main loop stays responsive. And the third tool is child process, spawn, exec, or fork, for running separate programs or Node scripts and talking to them over IPC or streams. The trap is reaching for cluster to fix a blocked loop, forking more processes just gives you more processes that each block the same way, the rule of thumb is I/O-bound means cluster, CPU-bound means worker threads.
