INTERVIEWER: Welcome in, thanks for making the time today. Let's run a quick API testing round, the kind of rapid technical screen you'd get for an SDET role. I'll ask a question, you give me your best answer, and I may push a little deeper here and there. Nothing tricky, just the fundamentals done well. Ready when you are.

CANDIDATE: Sounds great, let's do it.

INTERVIEWER: Let's start with a classic HTTP question. Walk me through the difference between PUT and PATCH, and tell me why it actually matters when you're testing.

CANDIDATE: Sure. PUT replaces the whole resource, so whatever I send becomes the complete new state, and any fields I leave out effectively get reset. PATCH is a partial update, it only changes the fields I include and leaves everything else untouched. The other key difference is idempotency, PUT is idempotent by definition, sending the same full replacement five times leaves the resource in the same state, while PATCH usually behaves that way but it's not guaranteed by the spec. From a testing perspective this matters a lot, because a really common bug is a PUT that silently wipes fields the client didn't send, so one of my go-to test cases is sending a PUT with a missing optional field and verifying the server resets it rather than keeping the old value, and then the mirror test for PATCH, confirming untouched fields survive the update.

INTERVIEWER: Nice, I like that you tied it back to an actual test case. Here's another one that comes up in almost every screen, what's the difference between a 401 and a 403?

CANDIDATE: This is the identity versus permission distinction. A 401 means unauthorized in name but what it really means is unauthenticated, the server doesn't know who I am, so I get it when the token is missing, malformed, or expired. A 403 means the server knows exactly who I am, my credentials are valid, but I'm not permitted to do this particular thing, say a regular user calling an admin endpoint. The memory hook I use is that 401 is the server asking who are you, and 403 is the server saying I know who you are and the answer is still no. In practice I always test both paths separately, no token should give me 401, and a valid token with the wrong role should give me 403, because if an API returns 401 for a permissions problem, that's a real defect worth raising.

INTERVIEWER: Good, and that's exactly the trap, people blur those two together. Let's zoom out a bit. When you get a response back from an API, what do you actually validate beyond the status code?

CANDIDATE: The status code is just the headline, so I work through several layers. First the response body, are the values correct and are the data types correct, an ID that comes back as a string when it should be a number is a bug even if everything reads fine to the eye. Second, the schema, I validate the overall structure against a JSON schema, required fields present, correct shape, which is much more maintainable than field by field assertions on a large payload. Third, headers, things like Content Type, caching headers, and rate limit headers. Fourth, response time, is it within the SLA, because a correct answer that takes 8 seconds is still a failure. And finally side effects, if I POST a new user I don't just trust the 201, I follow up with a GET to confirm the resource actually persisted with the right data. That last one is what separates checking the response from actually testing the system.

INTERVIEWER: That follow-up GET is a detail a lot of candidates miss. Let's build on that. How do you approach testing an endpoint that requires authentication, both in terms of framework design and coverage?

CANDIDATE: On the framework side, I acquire the token once in a setup phase or fixture, typically through a client credentials flow for OAuth or a login call for JWT, then I attach it as an Authorization Bearer header and reuse it across the suite, refreshing it only when it expires. Logging in fresh inside every single test is slow and adds noise, so caching the token is the pattern I reach for. But the setup is only half the answer, the coverage is where the real testing happens. I always include the negative auth cases, no token at all should return 401, an expired or tampered token should return 401, and a valid token that belongs to a user without the right role should return 403. Auth is a security boundary, so I'd argue those negative cases matter even more than the happy path.

INTERVIEWER: Agreed, the negative cases are where the risk lives. Now, you mentioned idempotency earlier when we talked about PUT. Define it properly for me, and give me the subtlety that trips people up.

CANDIDATE: Idempotency means that making the same request once or making it many times leaves the server in the same state. GET, PUT, and DELETE are idempotent, POST is not, because two identical POSTs typically create two separate resources. The subtlety, and this is the trap I'd flag, is that idempotency is about server state, not about getting an identical response. If I DELETE a resource twice, the first call might return a 200 and the second a 404, and that's still perfectly idempotent, because the end state of the server is the same either way, the resource is gone. Candidates often say idempotent means you always get the same response, and that's the wrong definition. As a tester I verify this directly, I send the same PUT or DELETE twice and assert that the state, not necessarily the status code, is unchanged after the repeat.

INTERVIEWER: That state versus response distinction is exactly what I was fishing for. Last one, and it's a design question rather than an HTTP question. Where do API tests sit in the test pyramid, and why would you invest there?

CANDIDATE: They sit in the middle layer, above unit tests and below UI tests, and I think it's the highest leverage layer for an SDET. The base of the pyramid is a large number of fast unit tests, the top is a small number of UI or end to end tests, and the API layer in between gives you the best balance of speed, stability, and coverage. API tests exercise real business logic and data integrity without any of the rendering or element waiting that makes UI tests slow and flaky, so they run in milliseconds rather than seconds and fail for real reasons. They also localize failures much better, when an API test breaks I know which endpoint and which contract, whereas a UI failure could be anything from the frontend down to the database. So my strategy is usually to push as much coverage as possible down into the API layer and keep the UI suite thin, just a few critical journeys.
