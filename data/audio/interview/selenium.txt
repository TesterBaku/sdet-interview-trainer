INTERVIEWER: Let's start under the hood. When your test code clicks a button, how does Selenium actually talk to the browser?

CANDIDATE: Selenium WebDriver drives a real browser through a browser-specific driver, so chromedriver for Chrome or geckodriver for Firefox, and my test code sends commands to that driver over the W3C WebDriver protocol, which is basically HTTP with JSON payloads. The driver translates each command into native browser actions. The reason this matters is that every single command is a separate HTTP round trip, client to driver to browser and back, and that round-trip model is the root of a lot of the latency and flakiness we end up managing with waits. A concrete example, when a click triggers a re-render and my very next command goes looking for an element, there's a real time gap between those two requests where the page can change underneath me. Selenium 4 standardized fully on the W3C protocol and also added things like relative locators and access to the Chrome DevTools Protocol.

INTERVIEWER: Good, and that round-trip point sets up my next question nicely. Selenium gives you implicit, explicit, and fluent waits. Which do you actually use, and why?

CANDIDATE: I use explicit waits almost exclusively, so WebDriverWait with ExpectedConditions, because they wait for a specific condition on a specific element, like the submit button becoming clickable or a spinner becoming invisible. An implicit wait is a global setting that polls the DOM on every element lookup, which sounds convenient but it's blunt, it applies everywhere and can quietly mask real timing problems. A fluent wait is just an explicit wait with a custom polling interval and a list of exceptions to ignore, and I reach for it when I need that finer control, say polling a slow grid every two seconds while ignoring stale element exceptions. The trap I always call out is mixing implicit and explicit waits, the two timers interact unpredictably and you get cumulative timeouts far longer than you intended, so I pick explicit and set the implicit wait to zero. And Thread dot sleep never goes in, it's a blind fixed wait that's slow when it's too long and flaky when it's too short.

INTERVIEWER: That's exactly the anti-pattern people miss. Now, locators. How do you choose between CSS selectors and XPath, and what makes a locator stable?

CANDIDATE: My default order is an id when it's unique and stable, because it's the fastest and least affected by DOM changes, then CSS selectors, and XPath only when I genuinely need its extra power. CSS is faster and more readable, while XPath can do things CSS can't, like matching on text content or walking up to a parent or ancestor, but it's slower and more brittle, especially positional XPath that breaks the moment the layout shifts. The deeper why is stability, I anchor to attributes that carry meaning across releases, and where I can I work with the front-end team to add test id attributes to interactive elements. On one dashboard we had auto-generated ids like btn underscore one two three four five that changed every build, so we added a data test id for the export button, swapped the locators over, and the flakiness on that page dropped to zero.

INTERVIEWER: Good, stable locators solve half the battle. Here's a scenario, your test intermittently throws a StaleElementReferenceException. What's going on and how do you fix it?

CANDIDATE: A stale element exception means I found an element earlier, cached the reference, and by the time I interact with it that node is no longer attached to the DOM, usually because the page re-rendered or navigated. The fix is not to retry blindly, first I figure out why the DOM is changing, then I re-find the element right before interacting instead of holding onto a cached reference, and I wait for the post-render state before touching anything. A real example, we had a React grid that refreshed after applying a filter, and the test clicked a row it had located before the refresh. We fixed it by waiting for the loading indicator to become invisible, then locating the row again, and it never flaked after that. Worth noting this is exactly the problem Playwright's lazy locators sidestep, because they re-query the DOM automatically on every action, whereas in Selenium that discipline is on me.

INTERVIEWER: Nice, and you anticipated a later question with that Playwright comment. Let's talk structure, why do teams use the Page Object Model, and how do you make a suite like that run in parallel?

CANDIDATE: Page Object Model means one class per page that encapsulates the locators and the actions on that page, so when the UI changes, the fix touches one file instead of fifty tests, and the tests themselves read like a user journey. Selenium ships PageFactory with the FindBy annotation as the classic helper, and a pattern I like is having navigation methods return the next page object, so the flow is typed and self-documenting. For parallel execution, the critical rule is that WebDriver is not thread-safe, so I never use a static shared driver, I use a ThreadLocal factory that gives each test thread its own driver instance, paired with TestNG's parallel configuration. Then Selenium Grid distributes those sessions across machines and browsers, hub and node classically, or router, distributor, and node in Selenium 4. The trap is a static driver field, it works fine right up until you turn parallelism on, and then everything collides.

INTERVIEWER: Strong answer, that static driver trap catches a lot of teams. Building on that, your suite runs in CI and some tests pass locally but flake there. How do you drive flakiness down systematically?

CANDIDATE: First I treat flakiness as a bug with a specific root cause, not weather, so instead of just raising timeouts I compare screenshots and WebDriver logs between passing and failing runs to see exactly what state differed. The usual suspects in CI are headless rendering differences, a smaller viewport, slower machines exposing timing gaps, and browser or driver version mismatches, so I reproduce in a container that matches the CI image. Prevention-wise, every interaction gets a condition-based explicit wait, tests own their data and share no state, and locators are stable ids or test ids. One case, a dropdown clicked fine locally but failed in CI as not interactable, and it turned out the CI viewport was smaller and the element sat below the fold, a scroll into view before the click fixed it permanently. On a six hundred test suite we also quarantined flaky tests so they ran without blocking the pipeline while we root-caused them, and the flake rate fell from fifteen percent to under two.

INTERVIEWER: Good, root cause over bigger timeouts, that's the senior move. Last one, you mentioned Playwright earlier, so give me the honest comparison. When would you pick Selenium and when would you pick Playwright?

CANDIDATE: The core difference is architectural, Selenium sends each command as an HTTP request through the WebDriver protocol to a separate driver binary, while Playwright holds a persistent connection to the browser in the style of the DevTools protocol, which makes it faster and lets it auto-wait, its actions automatically wait for elements to be ready, whereas in Selenium the explicit waits are my job. Playwright also has first-class network mocking and ships with bundled browsers, while Selenium's network interception is limited to CDP access and you manage driver binaries yourself. So why still Selenium, maturity and breadth, it supports basically any language and any browser, drives real browsers with high fidelity, and has huge grid and cloud ecosystem support, which matters for a large organization with an established stack. For a greenfield TypeScript project I'd seriously consider Playwright for speed and built-in reliability, but for a big cross-browser matrix on existing infrastructure, Selenium remains the safer bet, and the trap is treating this as religion instead of a trade-off you argue from requirements.
