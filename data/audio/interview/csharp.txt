INTERVIEWER: Let's start with the one that explains almost everything else. What's the difference between value types and reference types in C sharp?

CANDIDATE: Value types hold the data itself and get copied on assignment, while reference types hold a reference to an object on the heap, so assigning one just copies the reference and both variables point at the same object. That matters because it decides whether two variables can affect each other. So if I copy an int and change the copy, the original stays untouched, but if I copy an array variable and change an element through the copy, the original sees it, because there's only one array. Value types are things like int, double, bool, structs, and DateTime, while classes, strings, arrays, and delegates are reference types. The trap I always call out is passing a reference type into a method, the method can mutate the object and the caller sees it, but if the method reassigns the parameter to a new object, the caller is unaffected, because C sharp passes the reference itself by value.

INTERVIEWER: Good, and you caught the reassignment subtlety, which most people miss. Staying on design, when would you reach for an abstract class versus an interface?

CANDIDATE: I use an abstract class when types are genuinely a kind of the same thing and I want to share code and state, and I use an interface when unrelated types just need to promise the same capability. The reason is structural, a class can extend only one base class, so that slot is precious and should express an is-a relationship, while a class can implement many interfaces, so interfaces are perfect for can-do contracts. A concrete example from test code, I might have an abstract base page with shared navigation logic that every page object extends, and a separate interface like IDrawable or a searchable contract that only some pages implement. The abstract class can hold real implementations and properties that subclasses inherit, while the interface just says what members must exist. If I choose an abstract class where a contract was enough, I've burned the single inheritance slot for nothing, so I default to interfaces and reach for a base class only when there's real shared code.

INTERVIEWER: Nice, the single inheritance slot framing is exactly right. Now, strings. Why is string immutable in C sharp, and what does that cost you?

CANDIDATE: A string can never change once it's created, every operation like Replace or ToUpper returns a brand new string and leaves the original alone, which is why calling ToUpper without reassigning the result is a classic bug, the variable still holds the lowercase value. The why is safety, an immutable string is thread safe by nature, nobody can change a shared string underneath you, its hash never changes so it's safe as a dictionary key, and the runtime can intern identical literals into a shared pool. The cost shows up in loops, concatenating inside a loop creates a new object every pass, which turns into quadratic garbage. So the rule I follow is plain string for fixed or few values, and StringBuilder the moment I'm building a string across many iterations, appending as I go and calling ToString once at the end for a single allocation. One more nice property, the double equals operator on strings compares value, not reference.

INTERVIEWER: Solid. Let's go where the real depth is for a Playwright role. Explain async and await, and why it matters so much in your test code.

CANDIDATE: The async keyword marks a method that returns a Task, and await suspends that method without blocking the thread, the thread goes back to the pool while the I O completes and the method resumes when the result is ready. That's why it matters, it gives you concurrency without extra threads, a thread costs about a megabyte of stack, while tasks are cheap, so thousands of concurrent requests only need a handful of threads. For an SDET this is unavoidable because the Playwright API is async throughout, every click and navigation returns a Task. There are two traps I watch for. First, async void, it can't be awaited and its exceptions escape the caller, so I return Task everywhere except event handlers. Second, calling dot Result or Wait on a task can deadlock, so the rule is await all the way up. And when I want real concurrency, like firing several API requests at once, I start them all and use Task WhenAll to await them together.

INTERVIEWER: That's the answer I was hoping for, especially the await all the way up rule. Let's talk LINQ. What is deferred execution, and how does Single differ from First?

CANDIDATE: Deferred execution means a LINQ query doesn't run when you define it, it runs when you enumerate it, with ToList, a foreach, or Count. The why is that operators like Where and Select just build up a pipeline over an IEnumerable, nothing executes until something pulls results through it. The practical consequence is that the source can change between defining the query and enumerating it, so if I build a filter over a list and then add items before calling ToList, the new items are included, which surprises people and can also mean a query runs multiple times if you enumerate it twice. On Single versus First, Single asserts there is exactly one match and throws otherwise, while First just takes the first of possibly many. In tests I deliberately use Single when uniqueness is part of what I'm proving, say exactly one user with a given id, because First would quietly hide a duplicate. And the OrDefault variants return a default instead of throwing on empty, which is handy in assertions.

INTERVIEWER: Good, using Single as an assertion of uniqueness is a nice testing instinct. Next, exception handling. Walk me through try, catch, finally, and the difference between rethrowing with throw versus throw ex.

CANDIDATE: The try block wraps the code that might fail, catch handles a specific exception type, and I catch the narrowest type first, and finally always runs whether or not something was thrown, which makes it the place for cleanup like stopping a stopwatch. Worth noting C sharp has no checked exceptions, unlike Java nothing forces you to declare or catch anything, so discipline is on you. The rethrow difference is the classic trap, a bare throw statement rethrows the current exception and preserves the original stack trace, while throwing the caught variable resets the trace to that line, so when the failure surfaces in C I, the trace points at your catch block instead of the real cause. So I always prefer the bare throw, typically after logging. In my API tests the pattern is catch the HttpRequestException, log it with context, then rethrow bare so the report still shows exactly where the request actually failed.

INTERVIEWER: Right, and losing the stack trace in C I is exactly where that bites. One more in this area, what is the using statement for, given that C sharp already has a garbage collector?

CANDIDATE: The using statement gives deterministic cleanup, it calls Dispose automatically when the scope ends, even if an exception was thrown. The reason it exists alongside the garbage collector is that the G C frees managed memory on its own schedule, but it won't promptly release things like sockets, file handles, or browser contexts, and those you want closed the moment you're done. So in Playwright I write await using on a new browser context, and the context is disposed at scope exit no matter how the test finished, with await using and IAsyncDisposable handling the async cleanup. A real trap here is HttpClient, people wrap it in a using and create a new client per request, which leads to socket exhaustion, so the right move is to reuse one HttpClient across requests. The mental model I keep is memory belongs to the garbage collector, everything else, handles, connections, browser sessions, belongs to using and IDisposable.
