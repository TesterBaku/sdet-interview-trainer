INTERVIEWER: Let's start with a classic. What's the difference between WHERE and HAVING, and why does the difference exist at all?

CANDIDATE: The direct answer is that WHERE filters individual rows before any grouping happens, and HAVING filters the groups after GROUP BY has collapsed them, which is why HAVING can use aggregates like COUNT or SUM and WHERE can't. The reason the difference exists is the logical execution order of a query, the database runs FROM and the joins first, then WHERE, then GROUP BY, then HAVING, and only then SELECT, ORDER BY, and LIMIT. So when WHERE runs, the aggregate values simply don't exist yet. A concrete example I use all the time, if I want customers with more than a thousand in paid orders, I filter status equals paid in WHERE first so I only aggregate the right rows, then group by customer, then use HAVING on the sum. And that execution order explains a common trap too, a column alias defined in SELECT isn't visible in WHERE, because WHERE runs before SELECT, but it does work in ORDER BY, which runs last.

INTERVIEWER: Nice, and you got the execution order in there before I could ask. Next one, walk me through join types, and specifically, how would you find customers who have never placed an order?

CANDIDATE: A join combines rows from two tables on a matching condition, and the type decides what happens to non-matching rows. INNER keeps only rows that match on both sides, LEFT keeps every left row and fills the right columns with NULL when there's no match, RIGHT is the mirror of that, FULL OUTER keeps everything from both sides, and CROSS gives every combination, which is usually an accident. For customers with no orders, I'd LEFT JOIN orders onto customers on the customer id, then filter where the order id IS NULL, because those NULL rows are exactly the customers nothing matched. NOT EXISTS with a subquery works just as well. The trap I always call out is that if you filter the right table's columns in the WHERE clause, say where order status equals paid, you silently turn the LEFT JOIN back into an INNER JOIN, because the NULL rows fail that condition and get dropped. The fix is to put that condition in the ON clause instead.

INTERVIEWER: Good, and you already touched on NULL there, so let's stay on it. What makes NULL tricky in SQL, and how do you handle it?

CANDIDATE: The core idea is that NULL means unknown, not zero and not an empty string, and that unknown spreads through everything it touches. So you can never compare with equals NULL, that comparison is always unknown and never true, you have to use IS NULL or IS NOT NULL. Any arithmetic with NULL is NULL, five plus NULL is just NULL. And aggregates quietly skip NULLs, so AVG on a column ignores the null rows, and COUNT of a column counts only non-null values while COUNT star counts every row, which can surprise you in a report. For handling it, COALESCE returns the first non-null argument, so I can fall back from nickname to name to a default label, and NULLIF is handy for guarding a divide by zero, dividing by NULLIF of count and zero returns NULL instead of crashing. The nastiest trap is NOT IN against a subquery that returns a NULL, that query returns no rows at all, so I reach for NOT EXISTS instead.

INTERVIEWER: That NOT IN trap has bitten a lot of people. Now, suppose I ask you for the latest order per customer, or the second highest salary. What's your tool for those?

CANDIDATE: Both of those are window function problems. A window function computes over a set of related rows without collapsing them the way GROUP BY does, which is exactly what you need when you want the whole row back, not just an aggregate. For latest order per customer, I take ROW_NUMBER over a window partitioned by customer id and ordered by created at descending, wrap that in a subquery or CTE, and keep the rows where the row number is one. For second highest salary, same shape, DENSE_RANK over salary descending and filter where the rank equals two. The reason I pick each function deliberately is the tie behavior, ROW_NUMBER is always unique, RANK leaves gaps after ties, so you'd see one, one, three, and DENSE_RANK doesn't, one, one, two. For a salary question ties matter, so DENSE_RANK is the honest answer. The trap is forgetting PARTITION BY, then you're numbering the entire table as one big window instead of per customer.

INTERVIEWER: Solid, and the tie behavior is exactly the follow-up I was holding. Let's shift to performance. What does an index actually do for you, and when does it stop helping?

CANDIDATE: An index is a sorted lookup structure, usually a B-tree, that lets the database find rows without scanning the whole table. The trade is real though, reads on the indexed columns get faster, but every insert, update, and delete has to maintain the index, and it costs storage, so the guidance I give is to index what you filter, join, and sort on, not every column. A composite index on two columns follows the left prefix rule, it helps filters on the first column or on both, but not on the second column alone. The classic way an index stops helping is wrapping the column in a function, filtering where the year function of a date equals a value kills index use, and the fix is rewriting it as a date range so the predicate is sargable. When I suspect a slow query, I don't guess, I run EXPLAIN or EXPLAIN ANALYZE and check whether it's doing an index seek or a full scan.

INTERVIEWER: Good instinct going to the plan instead of guessing. One more on the database side, tell me about transactions and what ACID actually guarantees.

CANDIDATE: A transaction groups statements so they succeed or fail as one unit, and ACID is the set of guarantees that makes that reliable. Atomicity means all statements commit or none do, a rollback leaves nothing changed. Consistency means constraints hold before and after. Isolation means concurrent transactions don't corrupt each other, and durability means once it's committed, it survives a crash. The textbook example is a money transfer, begin a transaction, subtract a hundred from one account, add a hundred to the other, then commit, and if anything fails in between, the rollback means you never end up with money vanishing from one side only. The nuance worth adding is that isolation isn't one thing, it's a dial of levels trading consistency for concurrency, read uncommitted allows dirty reads, read committed is the common default, repeatable read stops non-repeatable reads, and serializable is strictest and stops phantoms. Knowing that dial exists matters when tests read data mid-transaction.

INTERVIEWER: Nice bridge to my last question, because you're interviewing for an SDET role, not a DBA role. How do you actually use SQL in your testing work?

CANDIDATE: For me SQL is a verification tool, I assert against the database, not just against the UI or the API. So after an action in a test, I query the table directly to confirm the row was written with the right values, which catches bugs where the API returns success but persists the wrong thing. I also run data quality checks, grouping by email with HAVING count greater than one to catch duplicates, and the LEFT JOIN with IS NULL pattern to catch orphaned orders pointing at customers that don't exist, plus null checks on required fields and row count reconciliation between source and target. Beyond assertions, I use SQL for setup and teardown, seeding fixtures and cleaning state from inside the test framework. And a real lesson learned, on an order history bug our test picked the latest row by max id while the app used the updated at timestamp, so backfilled records broke the assertion until we matched the query to the actual product rule.
