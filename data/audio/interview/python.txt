INTERVIEWER: Let's start with the classic. What's the difference between a list and a tuple, and what does mutability really mean in Python?

CANDIDATE: The direct answer is that a list is mutable and dynamic, while a tuple is immutable and hashable, which means a tuple can be a dict key or live in a set and a list can't. The deeper point is that in Python a variable is just a name bound to an object, not a typed box, and assignment binds names, it never copies. So if I write b equals a where a is a list, and then append to b, a changes too, because they're the same object. That aliasing bites people constantly in test code. If I need independence I take a shallow copy with a slice, or copy dot deepcopy when there are nested structures, since a shallow copy still shares the nested objects. And related to that, the is operator checks identity, the same object in memory, while double equals checks value equality, and mixing those two up is a classic trap.

INTERVIEWER: Nice, you got the aliasing point in there. Since you mentioned mutability biting people, tell me about the mutable default argument trap.

CANDIDATE: The trap is defining a function with something like a list as a default value. The direct answer is that default arguments are evaluated once, at definition time, not on every call, so that one list object is shared across every call that uses the default. The why is that the function object holds the default, and if it's mutable, every call mutates the same thing. Where this really hurts in testing is a helper like append result that collects values into a default list, because results quietly accumulate across tests and you get cross test pollution, where a test passes alone but fails in the suite. The fix is to use None as the sentinel default and create a fresh list inside the function body. It's a two line change, but interviewers ask it because it separates people who've actually debugged Python from people who've only read about it.

INTERVIEWER: Good, that cross test pollution angle is exactly why we ask. Next up, what's a generator, and when would you reach for one over a list comprehension?

CANDIDATE: A generator is a lazy iterator built with yield that produces values on demand, holding one item in memory at a time, while a list comprehension builds the entire list in memory up front. That memory difference is the whole point. For small collections I'll happily use a comprehension because it's readable, but for large or unbounded streams, like reading a huge log file line by line, a generator keeps memory flat no matter how big the input gets. A concrete SDET example is a generator that walks a paginated API, fetching a page, yielding each record, and only requesting the next page when the current one is exhausted, stopping when a page comes back shorter than the page size. The caller can stop consuming as soon as it finds the record it wants, so you never fetch pages you don't need. The trap in that pattern is breaking before yielding the last partial page, which silently drops the final records.

INTERVIEWER: I like that you tied it to pagination. Let's talk decorators, what are they and where have you actually used one?

CANDIDATE: A decorator is a callable that wraps a function to add behavior, things like logging, timing, or retry, without touching the function's body. It works because functions in Python are first class objects, you can pass them around and return them, and a closure captures the wrapped function inside the wrapper. The place I've genuinely used one is a retry decorator for flaky test utilities, where the outer function takes the max attempts and the exception types, returns a decorator, and the wrapper loops, calling the real function and re-raising the last exception only after all attempts fail. Two traps worth calling out. First, always apply functools dot wraps to the wrapper, otherwise the decorated function loses its name and docstring and introspection breaks, which matters to test frameworks. Second, only catch the specific exceptions you intend to retry, because a bare catch all will happily retry a genuine assertion failure and hide a real bug.

INTERVIEWER: Strong answer, and the point about retrying assertion failures is one people miss. Now the one everybody dreads, explain the GIL and what it means for running things in parallel.

CANDIDATE: The GIL, the Global Interpreter Lock, means only one thread executes Python bytecode at a time in CPython, so threads don't give you true multi core parallelism for CPU bound work. The reason it still matters which tool you pick is that the GIL is released during blocking I O, so threads overlap just fine on network calls, file reads, and browser actions, which is most of what test automation does. So my rule is threads or asyncio for I O bound concurrency, and multiprocessing for CPU bound work, since each process gets its own interpreter and its own GIL and can use a separate core. The practical example for an SDET is parallelizing a test suite, and there pytest xdist is the answer, because it runs tests across separate worker processes with dash n auto, which sidesteps the GIL entirely. The trap is assuming threading will speed up CPU heavy work, it won't, you just add lock contention.

INTERVIEWER: Good, and you landed on xdist which is the real world answer. Last one, walk me through pytest fixtures, how they work and how you use scopes.

CANDIDATE: A fixture is a function decorated with pytest dot fixture that provides setup, and pytest injects it into a test by matching the argument name, which is dependency injection without any wiring. If the fixture uses yield, everything before the yield is setup, everything after is teardown, and teardown runs even when the test fails. Scope controls the lifetime, function scope creates and tears down per test, then class, module, and session share progressively wider, so expensive things like a browser launch or an API client get module or session scope while anything that must stay clean per test stays function scoped. A real example is a create order fixture that calls the API to create an order, yields the order id to the test, and cancels it in teardown, so every test starts fresh regardless of execution order or parallelism. Shared fixtures live in conftest. The trap is calling the fixture function directly instead of declaring it as a parameter, which bypasses pytest entirely.
