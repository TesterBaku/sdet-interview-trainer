MAYA: We start with memory, because almost everything confusing in Java traces back to one distinction, primitives versus reference types. Here's the picture I want in your head. A primitive, like an int or a boolean or a double, is a sticky note with the value written directly on it, so when you assign it to another variable, Java photocopies the sticky note, and now you have two independent pieces of paper. If I write int a equals 1, then int b equals a, then change b to 9, the variable a is still 1, because b got a photocopy, not a connection.

LEO: Okay, and reference types are different how?

MAYA: A reference type, meaning objects, arrays, Strings, doesn't hold the thing itself, it holds the address of the thing, like a card with a street address written on it while the actual house lives on the heap. When you assign one object variable to another, Java photocopies the address card, not the house, so both variables now point at the same house, and if you walk in through either door and repaint the kitchen, everyone sees the new kitchen.

LEO: Which explains the array example that trips people up, right, where you copy an array into a second variable, change element zero through the copy, and the original array changes too, because there was only ever one array, and both variables were just holding directions to it.

MAYA: Exactly, and now the interview trap question resolves itself. Java is always, always pass-by-value, no exceptions, but here's the subtle part, for objects the value being copied is the reference itself. So when you pass a User into a method, the method gets a photocopy of the address card, which means it can drive to the house and change the fields, and those changes stick, but if the method tries to reseat that variable, pointing its copy of the card at a brand new User, the caller's variable never notices, because the caller still holds the original card.

LEO: So the memory hook is, you can repaint my house but you can't move me to a new one. Mutating the object works through the copied reference, reassigning the parameter does nothing outside the method.

MAYA: That's the line, and if you deliver it that cleanly in an interview, with the house analogy or without it, you've just separated yourself from half the candidate pool. And while we're in this neighborhood, know your wrapper classes too, every primitive has an object twin, int has Integer, boolean has Boolean, and you need them because generics and collections can only hold objects, so a List of Integer, never a List of int, and Java converts between them automatically, which is called autoboxing and unboxing.

LEO: Okay, chunk two, and I know where this is going, because no Java interview on earth skips Strings.

MAYA: None, because Strings hide two classic questions in one topic. First fact, String is a reference type but it's immutable, meaning once a String object is created it can never change, ever. Every operation that looks like a modification, toUpperCase, replace, substring, actually manufactures a brand new String and hands it back, leaving the original untouched. Think of every String as a stone tablet, you can't edit a stone tablet, you can only carve a new one, so if you call toUpperCase on a String and never assign the result to anything, you carved a beautiful new tablet and dropped it straight in the trash, and your original variable still says hello in lowercase.

LEO: Which is a real bug I have genuinely shipped, calling trim on a locator string and wondering why nothing changed, because I forgot to reassign the result.

MAYA: Everyone's done it once. Now, why did Java's designers make Strings immutable, because interviewers love asking for the why, not just the what. Three reasons worth naming. Immutable objects are automatically thread-safe, nothing can change under you mid-flight. They make safe HashMap keys, because the hash code can never shift after the key goes in. And they enable the String pool, where identical literals get stored once and shared, so two variables both assigned the literal hi actually point at the same pooled tablet.

LEO: Wait, so... that's why double equals sometimes works on Strings, isn't it. MAYA: And that's the deadly part, sometimes works, because a bug that fails every time gets caught in five minutes, but a comparison that passes with literals in your unit test and fails with real API data in the pipeline, that one eats an afternoon. So burn in the rule, double equals asks whether two variables point at the same object, the equals method asks whether the contents match, and for Strings and objects you always, always use equals. One more String beat before we move on, if you're building a big String in a loop, every concatenation carves a whole new tablet, which gets wasteful fast, so use StringBuilder, which is more like a whiteboard you keep appending to, and call toString once at the end for a single final allocation.

LEO: Chunk three, collections, which honestly is the stuff I use every single day in test code, parsing API responses into Lists and Maps.

MAYA: And the framework is simpler than people fear, because everything hangs off four ideas. LEO: And there's a follow-up lurking here that interviewers save for the end, the equals and hashCode question, which I'll admit felt like trivia until someone explained what actually breaks.

MAYA: So here's the physical picture. A HashMap is a coat check. MAYA: Records are the clean answer, and mentioning them signals you're current. Okay, trap corner, because a couple of wrong answers come up so often they deserve their own segment. LEO: Which is basically the worst possible outcome in our job, a test suite that lies.
