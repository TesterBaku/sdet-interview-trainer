MAYA: Okay, here's one that shows up in almost every SDET interview, usually in the first 5 minutes. The interviewer asks you what the difference is between a Docker image and a container, and then, before you can even finish, they follow up with why you'd use a container instead of a virtual machine. It sounds like a warmup question, but I've watched candidates fumble it in ways that ended the interview early.

LEO: And if your answer was going to be that an image is basically a small VM, well, this is the episode for you. I'm Leo.

MAYA: And I'm Maya. Today we're doing Docker for SDETs, which means images and layers, the container lifecycle, volumes, networking, Compose, and the part that actually wins the interview, which is using containers for clean, disposable test environments and CI.

LEO: So before we get into definitions, Maya, why does an interviewer even care whether a test engineer knows Docker? I'm not deploying production services, I'm writing tests.

MAYA: Because Docker is the cure for the oldest disease in our industry, the works-on-my-machine problem. When your test environment is a container, the image is the environment, so your laptop, your teammate's laptop, and the CI runner are all executing against byte-identical dependencies, the same browser version, the same driver, the same database. That means when a test fails, you're debugging the product or the test, not some drift in somebody's local setup, and for a hiring manager, an SDET who can say that with confidence is an SDET who can own the test infrastructure, not just write test cases.

LEO: Okay, so let's build it from the ground up. Container versus VM, what's the actual difference?

MAYA: Here's the analogy I always use, apartments versus houses. A virtual machine is a standalone house, it has its own foundation, its own plumbing, its own electrical system, because it virtualizes an entire operating system with its own kernel, and that's why a VM is gigabytes in size and takes minutes to boot. A container is an apartment in a big building, it has its own locked door, its own rooms, its own stuff, but it shares the building's foundation and plumbing, which in Docker terms means it shares the host operating system's kernel and only isolates the process, the filesystem, and the network. So a container is megabytes instead of gigabytes, it starts in seconds instead of minutes, and the runtime overhead is close to native.

LEO: So the memory hook is, apartment shares the building's foundation, house brings its own. Container shares the host kernel, VM brings its own kernel. And that's why I can spin up 10 containers on my laptop but 10 VMs would melt it.

MAYA: Exactly, and that lightness is the whole reason containers work for testing, because you can afford to create and destroy them constantly. Now, inside that world, the image versus container distinction. The image is a read-only template, built from a Dockerfile and tagged with something like myapp 1.2, and the container is a running instance of that image with a thin writable layer on top. The one-liner interviewers love is that the image is the class and the container is the object, one image, many containers.

LEO: Staying in your building analogy, the image is the architect's blueprint and each container is an actual apartment built from it, so I can stamp out 20 identical apartments from 1 blueprint, trash one, and build a fresh one that's exactly the same.

MAYA: Right, and images live in a registry, Docker Hub, ECR, GHCR, and you move them around with docker push and docker pull, the same way code lives in a git remote. Now here's the part that separates a mid-level answer from a senior one, and it's my second big analogy. An image isn't one solid block, it's a stack of layers, one layer per instruction in the Dockerfile, like the tiers of a wedding cake. Docker caches every tier, and here's the rule, if you change a tier, every tier above it has to be rebuilt, but everything below it comes straight from the cache.

LEO: Ah, so if the slow expensive tier is my dependency install, I want that near the bottom of the cake, under the stuff that changes all the time.

MAYA: That is exactly the interview answer. In a typical node Dockerfile you start from a base like node 20 alpine, then you copy only the package manifest files, then you run npm ci to install dependencies, and only after that do you copy the rest of your source code. Because the manifest rarely changes, the dependency layer stays cached, so an ordinary code edit rebuilds in seconds instead of re-downloading the entire dependency tree every single time.

LEO: Wait, so if I do it in the wrong order, if I copy all my source first and then install, then every time I touch 1 line of test code, Docker sees the copy layer changed, throws away everything above it, and reinstalls all my dependencies from scratch?

MAYA: Yes, and that ordering mistake is genuinely the number 1 cause of painfully slow builds that I see in real repos, so interviewers ask about it constantly, usually phrased as why is my Docker build slow or why does instruction order matter. While we're in the Dockerfile, two more quick ones that come up. CMD versus ENTRYPOINT, CMD sets the default command and it's easily overridden when you run the container, while ENTRYPOINT sets the fixed executable, with CMD supplying its default arguments. And multi-stage builds, where you compile or build inside a heavy image full of tooling, then copy only the finished artifact into a tiny runtime image, which gives you a smaller and safer final image. If they ask how to shrink a bloated image, your answer is a slim or alpine base, a multi-stage build, a dockerignore file so things like node modules and the git folder never enter the build context, and combining run steps to cut layers.

LEO: Let me lock in the everyday commands too, because screen-share interviews love these. Docker build with a tag makes the image, docker run creates and starts a container, docker ps lists what's running, docker exec with the interactive flags drops me into a shell inside a running container, docker logs shows me what it's printing, and docker stop and docker rm clean up.

MAYA: And 1 flag deserves special attention, port publishing. When you run with dash p 8080 colon 3000, you're mapping host port 8080 to container port 3000, host on the left, container on the right, and without publishing a port, the service inside the container is simply not reachable from your host at all, which matters because your test framework usually runs on the host and needs to reach the app in the container.

LEO: Okay, third chunk then, what happens to my data, and how do containers talk to each other?

MAYA: Here's my last analogy, a container is a hotel room. You can live in it, make a mess in it, but the moment you check out, housekeeping resets everything, because the container's writable layer is deleted with the container. If you want something to survive checkout, you don't leave it in the room, you put it in a storage unit outside the hotel, and that's a volume. A named volume is Docker-managed storage, the right choice for a database's data directory, while a bind mount maps a real folder from your host into the container, which is perfect for live-reloading your test code while you iterate. And the SDET twist is that the hotel room reset is usually a feature, not a bug, because for test data you actually want every run to check into a freshly cleaned room, a database with a known, clean state, so you often deliberately keep test data ephemeral, even using an in-memory tmpfs mount for the test database so it's fast and wiped on stop.

LEO: So bind-mount the code for fast iteration, but let the data die with the container so no test run inherits the last run's leftovers. What about networking?

MAYA: The key fact is that containers on the same user-defined network, which is what Compose creates for you automatically, reach each other by name, because Docker runs a built-in DNS. So your app container connects to the database at the hostname postgres on port 5432, using the service name as the address, like calling a coworker through the office directory by name instead of memorizing desk numbers that change daily.

LEO: And this is where people get burned by localhost, right?

MAYA: This is trap corner, yes, and it's the classic wrong answer of the whole topic. A candidate describes their Compose setup, and they say the app connects to the database at localhost 5432, or their tests inside a container hit the app at localhost 8080, and everything mysteriously refuses to connect. The trap is that inside a container, localhost means the container itself, not the host machine and not the other containers, because each apartment's front door only opens into its own apartment. The fix is to use the service or container name as the hostname, and in the rare case you genuinely need to reach the host machine from inside a container, there's a special hostname, host dot docker dot internal, for exactly that.

LEO: So the memory hook is, localhost is my own room, names are how I reach the neighbors. And there was a second trap you mentioned before the show, about depends underscore on.

MAYA: Right, the readiness trap. In a Compose file, depends on controls start order only, it does not wait for the dependency to actually be ready, so your database container has started but Postgres inside it is still initializing when your tests fire, and you get flaky connection failures on slow CI runners. The wrong answer is claiming depends on guarantees readiness, the right answer is that you add a healthcheck and have dependents wait for it, or build retry logic into the app or the test setup. Started is not the same as ready, and saying that sentence out loud in an interview scores real points.

LEO: So how does this whole topic actually surface in the room? Walk me through it as the ex-hiring-manager.

MAYA: It usually comes in 3 waves. Wave 1 is definitions, image versus container, container versus VM, what dash p does, quick and factual. Wave 2 is a scenario, they describe an integration test suite that needs a real database and ask how you'd set it up, and the winning shape is a Compose file with the app and a Postgres service, services talking by name on the shared network, a healthcheck gating the tests, and an ephemeral data directory so every run starts clean, all brought up and torn down with docker compose up and down. And wave 3, for stronger candidates, is Testcontainers, a library for Java, Python, dotnet, and Node that starts real containers from inside your test code, gives you a dynamic connection URL, and tears everything down after the tests, so you get real dependencies with the isolation of unit tests, no shared staging database, no cross-test contamination, no leftover state. Mention that unprompted and you sound like someone who has actually built this. And the CI angle rounds it out, GitHub Actions and GitLab can run jobs inside containers with version-pinned images, and official browser and Selenium Grid images give you consistent headless browsers on every run.
