LEO: Yeah, I'll admit that used to be my instinct too, because in the Selenium world a fresh browser was basically the only way to guarantee clean state, so you paid the startup tax and you suffered. MAYA: And I'm Maya, staff SDET, former hiring manager, and today we're doing Playwright, which is probably the hottest framework question in SDET interviews right now, because so many teams are either on it or migrating to it.

LEO: So before we answer your cold open, give people the 30-second version of what Playwright even is, because interviewers do ask for that framing.

MAYA: Sure, so Playwright is a modern end-to-end testing framework from Microsoft, it drives Chromium, Firefox, and WebKit through one single API, and it ships official bindings for C#, JavaScript and TypeScript, Python, and Java. But the part interviewers actually want to hear is how it talks to the browser, because instead of the WebDriver wire protocol that Selenium uses, Playwright speaks a DevTools style protocol, CDP style, over one persistent connection, which means one process holding an open line to the browser instead of shipping individual commands back and forth over HTTP.

LEO: And that architectural choice is why it can do things like auto-waiting and network interception natively, right, because it's sitting inside the browser's nervous system rather than knocking on the front door for every command.

MAYA: Exactly, and that's why this topic matters in interviews, because Playwright questions are never really trivia questions, they're architecture questions wearing a trench coat. When someone asks why Playwright is less flaky than Selenium, they're checking whether you understand what actually causes flakiness, and when they ask about contexts, they're checking whether you understand test isolation, which is a universal skill.

LEO: Okay, so let's answer the cold open, 300 tests, clean state, no 300 browsers. This is the browser, context, page model, right?

MAYA: It is, and here's the analogy I want stuck in your head, which is a hotel. The Browser is the hotel building itself, expensive to construct, so you build it once and you reuse it for the whole run. The BrowserContext is a hotel room, and here's the key part, every guest gets a freshly cleaned room with its own keycard, its own minibar, its own everything, and cleaning a room is cheap and fast compared to building a new hotel. And the Page is just a window in that room, the actual tab where you look out at the web and interact with it.

LEO: So the context has its own cookies, its own local storage, its own cache, completely walled off from every other context, even though they all live inside the same browser process.

MAYA: Right, it's essentially a fresh incognito profile that costs almost nothing to create, and that's the whole answer to the interview question. You launch one Browser, or one per worker, and then each test gets its own brand new Context, so you get perfect isolation, no leaked cookies, no shared login state, no test contaminating another, and you get it at the price of a room cleaning instead of a hotel construction project.

LEO: So the memory hook is, build the hotel once, clean the room every time, and the tab is just the window. And if I say in an interview that one Browser is reused while each test gets its own Context, I've basically nailed the isolation question.

MAYA: You have, and you can add the kicker, which is that this exact design is what makes safe parallelism possible, because Playwright runs multiple worker processes by default, each worker gets its own browser and its own contexts, nothing is shared, so tests can run at the same time without stepping on each other, and on CI you can even shard the suite across multiple machines.

LEO: Okay, chunk two, locators and auto-waiting, which I think is where most flakiness questions live. What's actually different about a Playwright Locator compared to grabbing an element in Selenium?

MAYA: So the big idea is that a Locator is lazy, it's not a reference to a specific DOM node you grabbed at some moment in time, it's more like a set of directions that get followed fresh every single time you act. Think of it like telling a taxi driver to take you to the tallest building downtown, rather than handing them a photograph of a building from last year. If the city rebuilds, the directions still work, because they're re-evaluated at the moment of the trip, and that's why Playwright basically doesn't have the stale element problem, the locator re-queries the DOM on every action.

LEO: And that connects to which locators you should prefer, because I know there's a priority order interviewers like to hear.

MAYA: There is, and the order is user-facing first. GetByRole is at the top, which finds elements by their ARIA role and accessible name, so a button named Sign in, and it's the most resilient because it matches how real users and assistive technology perceive the page, not how the markup happens to be structured this sprint. After that you have GetByLabel, GetByPlaceholder, and GetByText, then GetByTestId for stable data-testid hooks that the team controls, and only then CSS or XPath as a fallback, because those are coupled to markup that changes constantly.

LEO: So if the interviewer asks why GetByRole beats a CSS selector, the answer isn't that it's faster, it's that it survives redesigns because it targets meaning instead of structure, and as a bonus it nudges the team toward accessible UI.

MAYA: Perfectly said, and then auto-waiting is the second half of this chunk. Before Playwright performs any action, it automatically waits for the element to be attached to the DOM, visible, stable meaning it stopped animating, enabled, and actually able to receive events, and only then does it click. Same taxi analogy if you like, the driver doesn't drop you at the address while the building is still under construction, they wait until the door is open and reachable. And the twin feature is web-first assertions, so when you write expect locator to be visible, that assertion retries automatically until it passes or times out, no manual polling loop, no fixed delays.

LEO: Which brings us very naturally to trap corner, because I know exactly what the classic wrong answer is here. The interviewer describes a flaky test, something clicks too early or asserts too early, and the candidate proudly says they'd add a 2 second sleep to let the page settle.

MAYA: And that answer is a red flag in the shape of a solution, because a hard sleep is wrong in both directions at once, it's too long on fast days, so your suite burns minutes doing nothing, and it's too short on slow days, so the test still fails, just less often, which is actually worse because now it's intermittent. The dodge is to say that in Playwright you almost never wait for time, you wait for conditions, so you lean on auto-waiting for actions, you lean on web-first assertions for checks, and if you genuinely need to synchronize on something like a slow API call, you wait for the specific thing, for example page dot WaitForResponse for that endpoint, not for a number of seconds.

LEO: There's a second trap hiding nearby too, right, which is the candidate who says Playwright is basically just Selenium but faster.

MAYA: Yes, and that undersells the real difference, so the strong answer is that Selenium speaks the WebDriver protocol, where each command is a separate round trip and you assemble your own waits, your own mocking, your own tooling, while Playwright is a single API over a persistent CDP style connection with auto-wait, network interception, tracing, screenshots, and video all built in. It's not a faster horse, it's a different vehicle.

LEO: Okay, chunk three then, because you just mentioned network interception, and I feel like this is the one that separates mid-level from senior answers. How does mocking work and why do I care?

MAYA: So the scenario is that your UI test keeps failing, not because the UI is broken, but because the backend behind it is slow or flaky or returns different data every day, and the senior move is to cut the backend out of the picture. Playwright lets you do that with page dot Route, where you intercept any request matching a URL pattern and then fulfil it with your own response, modify it, or abort it entirely. The analogy here is a stunt double on a movie set, because when a scene is too dangerous or too unpredictable, you swap in a professional who delivers exactly the same move, take after take, identically. So you route the call to slash api slash users and fulfil it with a fixed JSON payload, maybe an empty list to test your empty state, and now the test is deterministic, it tests the UI and only the UI.

LEO: And the same isolation mindset shows up in how you handle login, which I know is a favorite speed question, because logging in through the UI before every single test is brutally slow.

MAYA: Right, and the pattern is called storage state, so you authenticate exactly once in a setup step, you save the cookies and local storage to a file, and then every new context loads that file at creation, which means every test starts already logged in, no login screen, no typing credentials 300 times. Back in our hotel, it's like the front desk pre-programming every room keycard before the guests arrive, one setup, and everyone walks straight in.

LEO: And when something does fail on CI, the debugging answer is the Trace Viewer, which honestly felt like magic the first time I used it.

MAYA: It really is the flagship debugging story, so you configure tracing to record on failure or on first retry, and when a test fails you open the trace and you get DOM snapshots for every step, the network calls, the console output, essentially a flight recorder for the test, and you scrub through the failure frame by frame without ever re-running it. Mention that alongside codegen, which records your clicks into a script with good locators, and retries configured centrally for known flaky areas, and you sound like someone who's actually operated a suite in CI.
