MAYA: And I'm Maya, staff SDET, former hiring manager, and today we're covering TestNG, the Java test framework that shows up in almost every Selenium-flavored interview loop. So if you truly understand TestNG, you can read JUnit 5, NUnit, and pytest at a glance, which means this 1 topic quietly covers 4.

LEO: That's actually why interviewers love it, right, it's a proxy question. They ask about TestNG but what they're really probing is whether you understand how test runners work at all.

MAYA: Exactly, and the name itself is a hint, the NG stands for Next Generation, because it was built to go beyond plain unit testing with flexible configuration, data-driven tests, groups, dependencies, and built-in parallelism, which is why it became the default runner in most Java plus Selenium stacks. So let's build it up in 3 chunks, starting with the lifecycle.

LEO: The annotations, this is the one they ask in the first 5 minutes of every screen.

MAYA: Right, and the mental model I want you to carry is a concert venue. Think about what happens on show night. The venue itself gets unlocked and powered up once for the whole evening, then the stage gets set for a particular act, then each band does its own setup, and then before every single song the crew resets the mics and the lighting. TestNG is exactly that, nested rings of setup from the widest scope down to the narrowest. At-BeforeSuite and at-AfterSuite run once around the entire suite, that's unlocking the venue. At-BeforeTest and at-AfterTest wrap a test tag inside the testng.xml file, that's setting the stage for one act. At-BeforeClass and at-AfterClass run once per test class, that's the band's own soundcheck. And at-BeforeMethod and at-AfterMethod run before and after every single at-Test method, that's the reset between songs.

LEO: So the order going in is suite, test, class, method, and then when everything tears down, the after hooks unwind in reverse, method first, suite last, like walking back out of the building the way you came in.

MAYA: That's the answer word for word, and notice how natural it is once you have the venue picture, because you'd never lock the whole building before the band finished packing up. In a Selenium context the classic pattern is at-BeforeMethod creates a fresh driver, the at-Test does its thing, and at-AfterMethod quits the driver, so every test gets a clean browser.

LEO: And the translation layer, since you promised this covers 4 frameworks. At-BeforeMethod is JUnit 5's at-BeforeEach, at-BeforeClass is roughly at-BeforeAll, NUnit calls them SetUp and OneTimeSetUp, and pytest does the same job with fixtures scoped to function, class, or session. Same lifecycle, different spelling.

MAYA: Perfect. Chunk 2 is assertions and data-driven testing, and these are 2 of the most reliable interview questions in existence. First, hard versus soft assertions. A hard assertion, your standard Assert dot assertEquals, fails fast, meaning the test stops dead at the first failed check. A soft assertion, using the SoftAssert class, collects every failure quietly and only reports them all together when you call assertAll at the end.

LEO: Give me the analogy, because I always blank on which is which under pressure.

MAYA: Think of a home inspector. The hard-assert inspector walks into the house, spots a crack in the foundation, turns around, and leaves, so you get a report with exactly 1 problem and no idea what else is wrong. The soft-assert inspector walks the entire house, notes the crack, the leaky faucet, the bad wiring, and hands you 1 complete report at the end. That final handover, that moment of handing you the full report, is the assertAll call, and here's the detail candidates miss, if you forget to call assertAll, nothing ever fails, the inspector just never files the report, and your broken test passes silently.

LEO: Ooh, that's a nasty one. So the memory hook is, soft assert without assertAll is an inspector who never mails the report. And when would I actually choose soft asserts?

MAYA: When you're validating several independent things in 1 test, say 5 fields on a profile page, and you want to see all the broken ones in a single run instead of playing whack-a-mole 1 failure at a time. And the mapping, JUnit 5 has an assertAll method, NUnit has Assert dot Multiple, and AssertJ and Playwright both ship soft assertions, so the concept travels everywhere. Now, staying in this chunk, data-driven tests, because it's the same theme of 1 test body, many checks. TestNG gives you 2 doors. Door 1 is at-DataProvider, a method that returns a 2-dimensional Object array, and TestNG runs your test once per row, so if your provider returns a valid login row and an invalid login row, the same test method executes twice with different inputs. Door 2 is at-Parameters, which injects simple values from testng.xml, things like a browser name or a base URL.

LEO: So the way I'd keep those straight is, DataProvider is for the actual test data, the rows, the interesting combinations, stuff that lives in code or gets computed, while Parameters is for environment-ish knobs that ops or CI might want to flip without recompiling anything.

MAYA: That's exactly the split, and if the interviewer pushes on other frameworks, JUnit 5 does this with at-ParameterizedTest plus sources, NUnit uses TestCase attributes, and pytest uses parametrize. Okay, chunk 3, and this is where TestNG really earns its keep in interviews, groups, dependencies, and parallel execution, all orchestrated through testng.xml. And my analogy for testng.xml is an air traffic control tower. The planes are your test classes, and the code inside them doesn't know or care about scheduling, but the tower decides everything about the traffic, which flights take off tonight, meaning which classes and groups get selected, what instructions each pilot receives, meaning the parameters injected, and critically, how many runways operate at once, meaning the parallel mode and the thread count. And the beautiful part is that the tower changes the schedule without rebuilding any planes, you reconfigure the whole suite without recompiling code.

LEO: So when someone asks what testng.xml is, the answer is, it's the suite descriptor, it selects classes and groups, passes parameters, and configures parallelism, all without touching or recompiling the tests themselves.

MAYA: Nailed it. Now the pieces the tower controls. Groups first, you tag a test with at-Test groups equals smoke, and then from testng.xml or the command line you run just that group, which is how teams carve a fast smoke slice out of a big regression suite. That's the same idea as JUnit 5 tags, NUnit categories, and pytest markers. Then priority, at-Test priority equals 1, which simply orders independent tests, lower numbers run first and the default is 0. And then dependencies, dependsOnMethods and dependsOnGroups, which say this test only makes sense if that one passed, and if the prerequisite fails, the dependent test gets skipped rather than failed.

LEO: Which brings us right back to your cold open. CreateUser fails, deleteUser gets skipped, and that is not a bug, that's the whole point, because deleteUser failing would be a misleading second failure, there was never a user to delete, so the skip is TestNG telling you the truth, only 1 thing is actually broken.

MAYA: Exactly right, and that sets up trap corner, because there are 2 classic wrong answers buried in this area. Trap number 1 is confusing priority with dependsOnMethods. Under pressure, candidates say they both just control test order, and that answer costs points, because priority is cosmetic ordering of independent tests, nothing more, while dependsOnMethods creates a real prerequisite relationship that can skip dependents when the prerequisite fails. One is a seating chart, the other is a load-bearing wall.

LEO: And trap number 2, I'm guessing, is the over-eager candidate who hears all this and starts bragging about their beautiful 10-test dependency chain where login feeds into search which feeds into checkout.

MAYA: That's the big one, and it's sneaky because it sounds like sophistication. Heavy use of dependsOnMethods couples your tests together, so 1 upstream failure cascades into a wall of skips, your report gets murky, and worse, chained tests can't run in parallel, because parallelism assumes tests are independent. So the strong answer is, I know dependencies exist, I use them sparingly for genuine setup ordering, and I prefer independent, isolated tests, and you can even add that JUnit deliberately limits ordering features for exactly this reason. Saying that sentence tells the interviewer you've maintained a real suite, not just read the docs.

LEO: So dodge the trap by naming the trade-off before they do. And that flows straight into parallelism, since you just said independence is what unlocks it.

MAYA: Right, so in testng.xml you set parallel to methods, tests, classes, or instances, plus a thread-count, say 4, and TestNG fans your suite out across threads. But, and this is the follow-up interviewers always spring, parallel tests must be thread-safe, they can share no mutable state, and in Selenium specifically the classic answer is a ThreadLocal wrapper around the WebDriver, so each thread gets its very own driver instead of 4 threads fighting over 1 browser. Back to the airport, 4 runways only work because each plane has its own pilot, its own fuel, its own instruments, and the moment 2 flights share a pilot the whole schedule collapses.

LEO: ThreadLocal means every thread gets its own car keys, nobody shares a steering wheel. Got it. What about the last mile, reporting and the extension points?

MAYA: Quick tour, because it comes up as a how-would-you question. Listeners hook into the test lifecycle, so implementing ITestListener and overriding onTestFailure is the canonical way to capture a screenshot on failure and attach it to the report, and you register the listener with the at-Listeners annotation or in testng.xml. For flaky tests there's IRetryAnalyzer, which re-runs a failed test a set number of times, and the senior move is to immediately add that retries should be used sparingly because they mask real flakiness rather than fixing it. And for reporting, TestNG emits HTML and XML out of the box, but real teams usually layer Allure or ExtentReports on top for richer output. Same extension idea exists everywhere, JUnit 5 has TestWatcher, pytest has hooks in conftest.
