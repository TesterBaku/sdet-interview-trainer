MAYA: Okay, here's a good one to start with. You call DELETE on the same order twice, and the first call comes back 200, but the second one comes back 404. So your teammate says, see, different responses, which means DELETE isn't idempotent. It sounds convincing. But is your teammate actually right?

LEO: And if you just said yes, well, this is the episode for you. I'm Leo.

MAYA: And I'm Maya, and this is the topic that opens almost every SDET interview I've ever run, which is API testing. Because here's the thing, if you can only test through the browser, you're really just testing the paint on the house, when APIs are the plumbing underneath.

LEO: And interviewers love it because it's so fast to probe, one question about status codes and they know your depth in thirty seconds, so let's build that depth. Start with why API tests even matter, because why not just do everything through the UI?

MAYA: Three reasons, and this is a spoken answer you should have ready. The first is speed, because an API test runs in milliseconds while a UI test has to spin up a whole browser. The second is stability, so no flaky selectors, no waiting for spinners, and far less flakiness overall. And the third is precision, because when a UI test fails, the bug could be anywhere in the stack, but when an API test fails, you know exactly which endpoint and which field. And that's why they sit in the middle of the test pyramid, more of them than UI tests, fewer than unit tests.

LEO: Okay, so the alphabet of this whole topic is HTTP methods, GET, POST, PUT, PATCH, and DELETE, and everyone knows the one-liners. So where do people actually get burned?

MAYA: Idempotency, and that word decides who gets the follow-up round, so let's really nail it. An operation is idempotent if calling it once, or calling it fifty times, leaves the server in the same state.

LEO: Okay, give me the physical version.

MAYA: Think of a light switch that only turns off. You press it once and the light's off, and then you press it twenty more times... and it's still off. The state of the room didn't change after that first press, and that's idempotent. Now compare that to dropping a letter in a mailbox, because if you drop two letters, there are two letters in the box, and that's POST.

LEO: So POST creates, and it's not idempotent, because two calls means two resources, which is exactly why double-clicking a place order button can charge you twice.

MAYA: Exactly, and that's a genuinely great example to use in an interview, because it shows you know why this matters. Now for the rest of the family. GET retrieves, and it's idempotent, it's safe, and it's cacheable. PUT replaces the entire resource, so it's idempotent too, because replacing the whole thing with the same thing twice is still the same thing. DELETE is idempotent as well. And then PATCH is the tricky one, because it's a partial update, and it's not necessarily idempotent.

LEO: Wait, hold on, PUT versus PATCH, say that again slower, because I know candidates blur these two.

MAYA: Okay, picture a whiteboard with your profile written on it. PUT erases the whole board and rewrites everything from scratch, so if you send the same PUT twice, the board reads identically. PATCH walks up and edits one line, say just the email. Now, if the patch says set the email to this value, doing it twice is fine, but if the patch says add five dollars to the balance, well, do that twice and you've got a different state. And that's why PATCH is only sometimes idempotent, because it depends on what the patch actually does.

LEO: Got it. So now back to the cold open, DELETE twice, 200 and then 404, and my teammate says different responses means it's not idempotent.

MAYA: And your teammate is wrong, and this is the exact nuance interviewers fish for. Idempotency is about the server's state, not the response you get back. After the first DELETE, the order is gone, and after the second DELETE, the order is... still gone. Same end state. That 404 just means there's nothing here to delete, which is honest, because the switch is already off.

LEO: So the sentence to memorize is, idempotent means same state after one call or many, not same response. Love it. Okay, status codes, give me the map, but the version for the ear.

MAYA: There are four families you care about. The 200s mean success, the 300s mean redirect, as in go look over there, the 400s mean the client messed up, so you sent something wrong, and the 500s mean the server messed up, meaning you did nothing wrong, the backend just fell over. And as a tester, that fourth-versus-fifth distinction is triage gold, because a 4xx points at the request while a 5xx points at the service.

LEO: And now we're in trap corner, because the single most asked status question in QA interviews is 401 versus 403.

MAYA: So here's the analogy that makes it stick, and it's a nightclub bouncer. A 401 means the bouncer has no idea who you are, maybe you didn't show ID, or your ID is fake or expired, so you're not authenticated. A 403 means the bouncer checked your ID, knows exactly who you are... and you're still not on the list. So you're authenticated, but not permitted.

LEO: So 401 is the bouncer asking who you are, and 403 is the bouncer saying I know exactly who you are, and the answer is still no.

MAYA: And that's the whole answer, delivered in one breath. Two more worth having ready, though. There's 429, which means too many requests, so you've hit a rate limit, and by the way, that's something you should be testing on purpose. And there's 409, which means conflict, and the classic case is trying to create a user with an email that already exists, where the right answer is often a 409, not a 500, and knowing that shows design maturity.

LEO: Okay, here's where I want to push, because I've seen this in real code review. Someone writes a test that calls the endpoint, asserts the status equals 200, and calls it done. Then the interviewer asks what you validate in an API test, and if your answer is just the status code, you're finished. So what's the real checklist?

MAYA: There are five layers, and I'd literally count them on your fingers in the room. First is the body, the actual values and the types, so is the price really 49.99, and is it a number, not the string 49.99, because type bugs are sneaky. Second is the structure, meaning does the response match the expected schema, with all the required fields present and nothing missing. Third is headers, so content type, caching, and security headers. Fourth is response time, because is it inside the SLA, the service level agreement, and a correct answer that takes nine seconds is still a failing test. And fifth, the one that separates seniors from juniors, is side effects.

LEO: Meaning the API said it created something, but did it actually create anything?

MAYA: Right, because a 201 from a POST is a claim, not proof. So you follow up with a GET and confirm the resource really persisted with the right data, which is trust, but verify. And if an interviewer hears you say you verify the side effect with a follow-up GET, they relax visibly. I've watched it happen.

LEO: Let's talk test design, because how would you test this endpoint is basically guaranteed to come up. So say I've got POST slash users, walk me through it.

MAYA: Positive cases first, so valid input, expect the 201, verify the body, and verify persistence. Then comes the negative catalog, and this maps beautifully onto the status codes we just did. Missing or malformed required fields, expect a 400 or a 422. A bad or expired token, that's a 401. A valid token but the wrong role, like a regular user hitting an admin endpoint, that's a 403. Asking for a user that doesn't exist gives you a 404, and creating a duplicate gives you a 409. And then boundary values, so empty strings, maximum lengths, zero, and negative numbers.

LEO: And what about the ones people forget?

MAYA: Pagination, so what happens on page zero, or a page past the end. Rate limiting, as in do you actually get the 429, and does it recover afterward. And concurrency, which is two requests updating the same resource at the same instant, and mentioning concurrency unprompted is a real senior signal.

LEO: Okay, auth, because none of these tests run if you can't get in the door. Give me the landscape.

MAYA: There are three tiers, from simplest to richest. First there's an API key, which is a static secret in a header, kind of like a building keycard. Then there are Bearer tokens, usually a JWT, a JSON Web Token, which is three parts separated by dots, a header, a payload, and a signature. The payload carries claims, so things like exp, which is the expiry time, sub, the subject, meaning who you are, and often your roles. And then there's OAuth 2.0, the full framework for delegated access, which is how log in with Google actually works.

LEO: And the interview favorite here, and I've been asked this twice, is token expiry. So your suite runs forty minutes, but your token dies at thirty. What do you do?

MAYA: Well, the wrong answer is to log in before every test, because that hammers the auth server and slows everything down. The right answer is to get the token once, in a setup step or a fixture, cache it, share it across tests, and refresh it only when it expires, either by checking that exp claim, or by catching a 401 and re-authenticating once. Because login per test is the mark of a suite that was never reviewed.

LEO: Last big idea, schema validation versus contract testing, and I'll be honest, for a long time I thought those were the same thing.

MAYA: Most people do, which is exactly why it gets asked. Schema validation checks the shape of one response, in isolation, so right fields, right types, on my side. Contract testing, and the big tool here is Pact, verifies the agreement between two services, so the consumer writes down exactly what it depends on, and the provider's build replays that contract and fails if it breaks.

LEO: So schema is me checking my own mailbox, and contract is both parties signing the lease, so if the landlord changes the terms, an alarm goes off.

MAYA: Perfect. And the payoff line for the interview is that contract tests catch breaking changes between services without needing a full end-to-end integration environment, so they're consumer-driven, fast, and cheap. That's the whole pitch of Pact in one sentence.
